# 2026 H1 电影化述职报告 V3 实施计划（无 API Key）

## 目标

在不依赖 Google Maps / Google Earth API Key 的前提下，重做太空开场、空间跃迁、澳门夜景城市、长距离沿街飞行与章节停靠构图。

核心验收标准：

1. 地球必须可辨认亚洲、海洋、城市灯光与大气层，禁止纯色球体。
2. 澳门城市必须基于真实海岸线、道路与建筑 footprint，夜景材质可艺术化，但地理结构不能随机。
3. 每段飞行 8–15 秒，至少包含 2–3 次转弯、明显前景视差和目标建筑逐渐出现。
4. 开场取消体积云，改为虫洞 / 空间折叠，跃迁后在澳门上空停留，再俯冲进入城市。
5. 航线、机体朝向、镜头视线、地标构图四层独立；禁止原地大角度调头、望向空白与穿楼。
6. 保留报告 UI、章节内容和 GitHub Pages 子路径部署。

## 技术路线

- 渲染：Three.js，静态 GitHub Pages。
- 地球：公开可再分发地球纹理 + 夜间城市灯光纹理 + 云层纹理 + 自定义大气层 Shader；不调用地图 API。
- 澳门数据：预处理后的 OpenStreetMap 道路、建筑 footprint、海岸线与水体，作为本地 JSON 随页面发布。
- 建筑：真实 footprint 拉伸；按 OSM height / building:levels 推算高度；缺失高度按用途和 footprint 规则估算。
- 夜景：程序化窗灯、楼顶灯、道路灯带、雾、Bloom；避免随机盒子城市。
- 地标：澳门塔、新葡京、大三巴、路氹酒店群、摩珀斯、机场分别使用独立 Hero Geometry / GLTF。
- 路线：基于真实道路图计算预设路线；离线生成 Catmull-Rom 航线与安全走廊，不在浏览器运行时调用 Overpass。
- 避障：路线生成阶段检测建筑包围盒 / 多边形；运行时保留审计接口。
- 虫洞：全屏 Shader，包含径向扭曲、星线拉伸、色散、中心折叠、短黑场和澳门上空出洞。

## 分阶段实施

### Phase 0：基线与回退

- 保留现有 V2，不直接覆盖。
- 新建 V3 入口与独立模块。
- 页面提供 `?version=v2` / V3 路径对比。

验收：旧版仍可访问；V3 空壳可加载。

### Phase 1：真实地球与虫洞开场

- 删除纯色 Sphere 地球。
- 接入真实地球昼夜纹理、大气辉光和澳门定位。
- 删除球状云层。
- 新增虫洞跃迁与澳门上空出洞停留。

验收：开场地球有大陆、海洋、灯光；跃迁无棉花云；澳门上空停顿清晰。

### Phase 2：真实澳门城市底图

- 导入本地 OSM 数据。
- 生成海岸线、水体、道路、桥梁和真实 footprint 建筑。
- 夜景材质分层：住宅、商业、酒店、公共建筑。
- 添加路灯、窗灯、雾与水面反射。

验收：俯视时澳门半岛、氹仔、路氹与机场结构可辨认；不再是随机盒子阵列。

### Phase 3：地标 Hero Asset

- 重建 6 个章节地标轮廓与夜间灯光。
- 地标与真实坐标、道路和周边街区对齐。
- 设置地标 LOD 与独立灯光。

验收：每个章节不看文字也能识别目标建筑；建筑处于画面右下安全区。

### Phase 4：长距离沿街飞行

- 构建道路图与预设章节路线。
- 每段路线扩展为 8–15 秒。
- 增加道路转弯、街区穿行、桥下 / 建筑间通道等空间事件。
- 机体朝向使用航线切线；镜头使用前方预瞄；最后 12% 才渐进构图地标。
- 移除无意义 departure / settling 静止。

验收：全程不穿楼、不突然掉头、不长期望向海面或空白；目标建筑由远及近出现。

### Phase 5：电影质感与性能

- 加入轻量 Bloom、速度拖影、动态曝光、镜头震动和运动音效接口。
- 建立桌面高画质 / 低画质两档。
- 加入加载进度、错误提示和 WebGL 能力检查。

验收：1080p 演示机稳定；页面加载失败时有清晰提示；低画质仍保持路线与内容完整。

### Phase 6：逐章视觉验收与部署

- 自动截图开场、出洞、每段飞行中点、每个停靠点与结尾。
- 输出构图 / 碰撞 / 空白视野 QA 数据。
- 合并到 `main` 后继续使用子路径发布，不覆盖仓库旧首页。

目标预览路径：

`/instashopping-prd/reports/2026-h1-cinematic-v3/`

## 不采用的方案

- 不使用 Google Photorealistic 3D Tiles：需要 API Key 与 Billing。
- 不嵌入 Google Earth 网页：无法作为可编程实时场景复用。
- 不继续修补随机盒子城市。
- 不再使用压扁球体模拟云层。
- 不依赖运行时 Overpass API，避免述职现场网络与限流风险。

## 执行顺序

严格按照 Phase 0 → 1 → 2 → 3 → 4 → 5 → 6 推进。每阶段完成后先保留可预览版本，再进入下一阶段，避免再次出现一次性重写后无法视觉验收的问题。
